# GEO 搜索能力诊断：设计方案

# 总览：执行步骤与讨论路线图

## 目标

给定一个开源社区及期望搜索的话题，自动完成主流 AI 平台的 GEO 搜索能力判断，并给出社区文档的改进建议。

## 执行步骤

```
步骤一：定义输入 → 步骤二：多平台搜索采样 → 步骤三：诊断分析 → 步骤四：改进建议输出
         │                   │                     │                    │
    社区名称+关键词        API 调用各平台         评分+对比+缺口       看板展示+Excel导出
```

---

### 步骤一：定义输入

**目标**：确定要监控的社区和搜索关键词。

**已讨论决策**：
- 关键词来源：社区运营代表根据调研结果手动输入 + 自动提取（Issue/论坛）双通道
- 更新频率：每周增量更新，每月全量聚类
- 语言：中英文双语
- 详细设计见 → 第二部分（第六~十一节）

**待讨论问题**：

| 编号 | 问题 | 状态 |
|------|------|------|
| 1.1 | Issue 数据范围：只看主仓库还是覆盖所有子项目？ | 待讨论 |
| 1.2 | openEuler 论坛是否有公开 API？若无，是否接受爬取？ | 待调研 |
| 1.3 | 是否接入 Google Search Console / 百度站长工具作为补充数据源？ | 待确认 |
| 1.4 | 关键词初始数量建议多少？（建议 MVP 阶段 20-30 个） | 待确认 |

**开发思路**：

```
1. 设计关键词 YAML 配置格式（手动输入入口）
2. 开发 Issue 采集模块
   - 对接 Gitee API / GitHub API
   - 按时间范围拉取 Issue 标题 + 标签
   - LLM 过滤非 GEO 相关 Issue（纯 Bug 类）
   - LLM 将技术标题改写为自然语言搜索词
3. 开发论坛采集模块
   - 调研论坛 API 可用性
   - 按浏览量/回复数排序提取热帖
4. 开发合并去重模块
   - Embedding 计算语义相似度
   - 相似度 > 0.85 的合并
   - 手动关键词优先级 > 自动提取
5. 开发双语翻译模块
   - LLM 中英互译
   - 只填一种语言时自动补全另一种
6. 开发定时任务
   - 每周 Cron 触发增量更新
   - 每月触发全量重聚类
```

### 步骤二：多平台搜索采样

**目标**：用关键词在各 AI 平台上搜索，采集原始结果。

**已讨论决策**：
- 采样方式：半自动（API 接入）
- MVP 平台：Perplexity + ChatGPT + DeepSeek
- 详细设计见 → 第一部分（第一~五节）

**待讨论问题**：

| 编号 | 问题 | 状态 |
|------|------|------|
| 2.1 | 各平台 API Key 如何管理？统一环境变量还是配置文件？ | 待确认 |
| 2.2 | 采样频率：每个关键词每周采样一次还是多次？多次可观察波动但成本更高 | 待确认 |
| 2.3 | Perplexity API 的 `search_recency_filter` 参数如何设置？ | 待调研 |
| 2.4 | ChatGPT API web search 返回的引用来源格式解析 | 待调研 |
| 2.5 | DeepSeek 无联网搜索，如何评估？仅测内置知识覆盖还是接入第三方搜索？ | 待讨论 |
| 2.6 | 是否需要记录 AI 回答原文用于人工复核？ | 待确认 |

**开发思路**：

```
1. 设计采样结果数据模型
   {
     keyword_id: "kw_001",
     platform: "perplexity",
     lang: "en",
     query: "openEuler installation guide",
     timestamp: "2026-03-10T08:00:00Z",
     raw_response: "...",          // AI 原始回答
     citations: ["url1", "url2"],  // 引用来源（Perplexity）
     mentions_community: true,     // 是否提及社区
     competitors_mentioned: ["CentOS", "Ubuntu"]
   }

2. 开发各平台采样适配器（统一接口，各自实现）
   - PerplexitySampler：调用 Perplexity API，提取 citations
   - ChatGPTSampler：调用 OpenAI API + web_search tool，解析引用
   - DeepSeekSampler：调用 DeepSeek API，仅测知识覆盖

3. 开发采样调度器
   - 输入：关键词库
   - 按语言选择对应平台
   - 控制并发和速率限制
   - 失败重试机制

4. 采样结果持久化
   - 存入 PostgreSQL
   - 保留历史记录用于趋势分析
```

### 步骤三：诊断分析

**目标**：对采样结果进行结构化评分和对比分析。

**已讨论决策**：
- 评分体系：提及分(0-3) + 引用分(0-3) + 准确分(0-2) + 排名分(0-2) = 总分(0-10)
- 综合评分：Perplexity×0.35 + ChatGPT×0.40 + DeepSeek×0.25
- 准确率由 LLM 自动判断（判断标准待后续讨论）
- 详细设计见 → 第三部分（第十二~十三节）

**待讨论问题**：

| 编号 | 问题 | 状态 |
|------|------|------|
| 3.1 | LLM 准确率判断标准：如何定义"正确"？对照什么基准？ | **待讨论（关键）** |
| 3.2 | 准确率判断是否需要提供社区官方文档作为参考上下文？ | 待讨论 |
| 3.3 | 评分的 LLM Prompt 模板如何设计？需要结构化输出 | 待设计 |
| 3.4 | 竞品列表是否可配置？不同社区的竞品不同 | 待确认 |
| 3.5 | 评分结果是否需要人工抽检校准？如何建立 LLM 评分的可信度？ | 待讨论 |

**开发思路**：

```
1. 设计 LLM 评分 Prompt
   - 输入：关键词 + AI 平台原始回答 + 社区官方信息（可选）
   - 输出：结构化 JSON，包含四个维度的分数和理由
   - 示例 Prompt：
     "你是一个 GEO 分析专家。以下是用户搜索'{keyword}'时，
      {platform}的回答。请评估该回答对{community}社区的覆盖质量，
      按以下维度打分并说明理由：..."

2. 开发评分引擎
   - 逐条对采样结果调用 LLM 评分
   - 解析结构化输出
   - 异常分数检测（偏离历史均值过大时标记复核）

3. 开发聚合分析模块
   - 按平台聚合：计算平台评分
   - 按关键词聚合：找出高分词和低分词
   - 按时间聚合：计算周环比变化
   - 竞品对比分析：统计竞品提及率

4. 开发内容缺口识别
   - 提及分 = 0 的关键词 → 标记为内容缺口
   - 引用分 < 2 的关键词 → 标记为引用缺失
   - 准确分 = 0 的关键词 → 标记为信息过时
```

### 步骤四：改进建议与输出

**目标**：将诊断结果可视化展示，生成可执行的改进建议，支持 Excel 导出。

**已讨论决策**：
- 输出形式：前后端分离的监控看板
- 支持 Excel 导出（评分总览 + 改进建议两个 Sheet）
- 改进建议按 P0-P2 分优先级
- 详细设计见 → 第三部分（第十四~十六节）

**待讨论问题**：

| 编号 | 问题 | 状态 |
|------|------|------|
| 4.1 | 前端框架选型：React 还是 Vue？ | 待确认 |
| 4.2 | 看板是否需要多社区支持？（一套系统服务多个社区） | 待确认 |
| 4.3 | 是否需要用户登录和权限管理？ | 待确认 |
| 4.4 | 改进建议是否需要对接任务管理系统（如 Gitee Issue 自动创建）？ | 待讨论 |
| 4.5 | 是否需要周报自动推送（邮件/企业微信/钉钉）？ | 待确认 |
| 4.6 | Excel 导出是否需要自定义时间范围和筛选条件？ | 待确认 |

**开发思路**：

```
1. 后端 API 设计（FastAPI）
   - GET  /api/dashboard/overview          → 总览数据
   - GET  /api/dashboard/platform/{name}   → 平台维度数据
   - GET  /api/dashboard/keyword/{id}      → 关键词维度数据
   - GET  /api/dashboard/trend             → 趋势数据
   - GET  /api/export/excel                → Excel 导出
   - POST /api/keywords                    → 手动添加关键词
   - GET  /api/suggestions                 → 改进建议列表

2. 前端看板页面
   - 首页仪表盘：综合评分、平台覆盖率、周趋势图
   - 平台详情页：各指标雷达图、关键词得分列表
   - 关键词详情页：跨平台对比、历史趋势、改进建议
   - 设置页：关键词管理、竞品配置、导出功能

3. 改进建议生成模块
   - LLM 根据评分 + 原始回答 + 缺口类型生成建议
   - 自动分配优先级（P0/P1/P2）
   - 建议去重（相似建议合并）

4. Excel 导出模块
   - Sheet 1：GEO 评分总览（关键词 × 平台矩阵）
   - Sheet 2：改进建议（优先级、诊断、建议、负责方向、状态）
   - 支持按时间范围、分类、优先级筛选

5. 通知推送（可选）
   - 每周评分报告自动生成
   - 推送到指定渠道
```

# 第一部分：主流 AI 搜索平台分析

## 一、平台分类

### 1.1 专用 AI 搜索引擎

| 平台 | 典型用户特征 | 判断依据 |
|------|-------------|---------|
| Perplexity | 技术从业者、研究者、英文信息消费者 | 产品定位为"答案引擎"，无中文界面，用户需有主动搜索习惯且偏好结构化引用 |
| Google AI Overview (SGE) | 最广泛的通用搜索用户 | 嵌入 Google 搜索结果页，用户无需主动选择，覆盖所有 Google 用户 |
| Bing Copilot | Windows/Edge 生态用户、企业用户 | 绑定 Edge 浏览器和 Windows 系统，企业 M365 用户自动接入 |
| 秘塔搜索 (Metaso) | 国内学术/知识型用户 | 主打"没有广告的搜索"，学术搜索功能突出，在知识社区传播 |
| 天工 AI 搜索 | 国内普通用户 | 昆仑万维推出，国内搜索市场补充 |

### 1.2 带搜索能力的 AI 助手

| 平台 | 典型用户特征 | 判断依据 |
|------|-------------|---------|
| ChatGPT | 全球最大 AI 用户群，偏技术和知识工作者 | MAU 超 2 亿，搜索功能默认开启，用户以提问方式获取信息 |
| 豆包 | 国内年轻用户、抖音生态用户 | 字节系产品，抖音导流，用户偏年轻化和泛娱乐 |
| DeepSeek | 国内技术社区用户、开源爱好者 | 开源模型出圈，技术社区口碑传播，开发者浓度高 |
| Kimi (月之暗面) | 国内知识工作者、学生 | 主打长文档理解，用户多为需要处理文档/论文的群体 |
| Claude | 海外开发者、技术写作者 | 定位专业级 AI，用户偏高端技术群体，无内置搜索但有联网能力 |

## 二、对 openEuler 社区的优先级判断

从开源操作系统社区的角度，按用户匹配度分层：

### 高优先级（技术用户集中）

- **Google AI Overview**：全球开发者的默认搜索入口
- **ChatGPT**：开发者问"怎么装 openEuler"最常用的地方
- **Perplexity**：技术选型对比场景（openEuler vs CentOS）
- **DeepSeek**：国内开发者首选

### 中优先级

- **Bing Copilot**：企业用户评估国产 OS 替代
- **Kimi**：用户上传 openEuler 文档后提问
- **秘塔搜索**：学术/技术调研

### 低优先级

- **豆包**：用户群与技术社区重叠度低
- **天工**：市场份额小

## 三、判断逻辑

1. **用户-产品匹配**：openEuler 的目标用户是开发者和企业 IT 人员，优先关注开发者浓度高的平台
2. **搜索场景**：操作系统相关问题多为"怎么做"（教程）和"选哪个"（对比），这类问题在 ChatGPT、Perplexity、DeepSeek 上最常见
3. **国内外分层**：openEuler 同时面向国内和国际市场，需要覆盖中英文两个生态
4. **API 可用性**：半自动方案需要 API 支持，这也是一个筛选条件

## 四、API 可用性现状

| 平台 | API 搜索能力 | 备注 |
|------|-------------|------|
| Perplexity | 有 API，返回引用来源 | 最适合 GEO 分析 |
| ChatGPT | 有 API，支持 web search tool | 可获取搜索结果 |
| DeepSeek | 有 API，但无联网搜索 | 只能测知识覆盖，不能测搜索引用 |
| Google AI Overview | 无公开 API | 需要浏览器自动化 |
| Kimi | 有 API，支持联网 | 可测试 |

## 五、MVP 阶段结论

MVP 阶段聚焦 **Perplexity + ChatGPT + DeepSeek** 三个平台，覆盖三个维度：

- **Perplexity**：海外搜索，返回引用来源，最适合分析内容是否被引用
- **ChatGPT**：全球通用，用户基数最大，代表主流 AI 搜索行为
- **DeepSeek**：国内技术社区，代表国内开发者的 AI 使用习惯

三者均有 API 支持，可实现半自动化采样。

# 第二部分：关键词定义策略

关键词系统分为**手动输入**和**自动提取**两条路径，最终合并为统一的关键词库。

## 六、手动输入

社区运营根据经验手动给定关键词，使用结构化格式：

### 输入格式

```yaml
community: openEuler
keywords:
  - topic: "openEuler 安装教程"
    category: 教程
    priority: high
    lang: zh
  - topic: "openEuler vs CentOS"
    category: 对比
    priority: high
    lang: en
```

### 分类维度

| 类别 | 用户意图 | 示例 |
|------|---------|------|
| 教程 | 怎么做 | "openEuler 安装"、"openEuler Docker 部署" |
| 对比 | 选哪个 | "openEuler vs CentOS"、"openEuler vs Ubuntu" |
| 故障 | 出问题了 | "openEuler yum 报错"、"openEuler 内核升级失败" |
| 特性 | 了解功能 | "openEuler A-Tune"、"openEuler iSulad" |
| 迁移 | 怎么切换 | "CentOS 迁移 openEuler"、"x2openEuler 使用" |

## 七、自动提取

### 7.1 从 Issue 提取

**数据源**：Gitee/GitHub 上社区各仓库的 Issue

**提取逻辑**：

1. 拉取近 N 个月的 Issue 标题 + 标签
2. 过滤掉纯代码 Bug（如"段错误""编译失败"等不适合 GEO 的话题）
3. 聚类相似 Issue，提取高频主题
4. 用 LLM 将技术性标题改写为用户可能搜索的自然语言查询

**转化示例**：

| 原始 Issue 标题 | 转化后的搜索关键词 |
|----------------|------------------|
| `yum install fails on openEuler 22.03` | "openEuler 22.03 yum 安装失败怎么办" |
| `Support for ARM64 Docker image` | "openEuler ARM64 Docker 镜像" |
| 多个 Issue 提到 `x2openEuler` 迁移问题 | "CentOS 迁移到 openEuler 教程" |

### 7.2 从论坛提取

**数据源**：openEuler 论坛帖子（forum.openeuler.org）

**提取逻辑**：

1. 按浏览量/回复数排序，取 Top N 热门帖
2. 提取帖子标题和标签
3. 按时间窗口计算话题热度趋势（上升/稳定/下降）
4. 用 LLM 归纳为搜索关键词

### 7.3 从搜索引擎反向提取（补充）

**数据源**：Google Search Console（如果社区有接入）、百度站长工具

**逻辑**：直接看用户已经在用什么关键词搜到了社区网站，哪些词展示了但点击率低（说明内容不够吸引人）。

## 八、合并与去重

两条路径的关键词汇入统一关键词库：

```
手动关键词 ──┐
             ├──→ 合并 → 去重/聚类 → 优先级排序 → 最终关键词库
自动关键词 ──┘
```

**去重策略**：
- 语义相似度 > 0.85 的合并为一个（用 Embedding 计算）
- 手动关键词优先级高于自动提取的
- 自动提取的按"出现频次 × 时间衰减"排序

## 九、定期更新机制

更新频率：**每周自动运行**，每月全量重新聚类。

### 每周增量更新流程

1. 拉取增量数据（上周新增 Issue + 论坛热帖）
2. 提取新关键词
3. 与现有关键词库合并去重
4. 标记变化（新增/热度上升/热度下降）
5. 输出更新后的关键词库 + 变更摘要

### 变更摘要示例

```
[2026-03-10 周报]
新增关键词：3 个
  + "openEuler 24.03 LTS 升级指南"（来源：Issue x5）
  + "openEuler AI 框架兼容性"（来源：论坛热帖）
  + "openEuler on RISC-V"（来源：Issue x3）
热度上升：2 个
  ↑ "CentOS 迁移 openEuler"（Issue 频次 +40%）
热度下降：1 个
  ↓ "openEuler 20.03 安装"（论坛关注减少）
```

### 增量 vs 全量策略

- **每周**：只处理增量数据，节省 API 调用
- **每月**：全量重新聚类，清理过时关键词，重新计算优先级

## 十、中英文双语方案

每个关键词同时维护中英文版本：

### 关键词存储格式

```yaml
- id: kw_001
  zh: "openEuler 安装教程"
  en: "openEuler installation guide"
  category: 教程
  source: manual
  priority: high

- id: kw_002
  zh: "CentOS 迁移到 openEuler"
  en: "migrate from CentOS to openEuler"
  category: 迁移
  source: auto/issue
  priority: high
  first_seen: 2026-03-03
  frequency: 12
```

### 双语生成方式

- 中文 Issue/论坛 → LLM 提取中文关键词 → LLM 翻译为英文
- 英文 Issue → LLM 提取英文关键词 → LLM 翻译为中文
- 手动输入支持只填一种语言，另一种自动补全

### 搜索采样时的语言对应

- **Perplexity / ChatGPT** → 用英文关键词
- **DeepSeek** → 用中文关键词
- 三个平台都可以双语各跑一轮，对比中英文覆盖差异

# 第三部分：输出形式——GEO 监控看板

## 十二、看板核心指标

分三个层级：

### 12.1 总览指标（首页仪表盘）

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| GEO 综合评分 | 社区在 AI 搜索生态中的整体表现 | 各平台加权平均分（0-100） |
| 平台覆盖率 | 多少个平台能搜到社区内容 | 有提及的平台数 / 总平台数 |
| 周趋势 | 本周 vs 上周的评分变化 | 分数差值 + 箭头 |

### 12.2 平台维度（按平台展开）

每个平台（Perplexity / ChatGPT / DeepSeek）独立评分：

| 指标 | 说明 |
|------|------|
| 提及率 | 搜索关键词后，回答中提到社区的比例 |
| 引用率 | 回答中引用了社区官方链接的比例（仅 Perplexity 可精确统计） |
| 准确率 | AI 给出的信息是否正确、是否过时（由 LLM 自动判断） |
| 排名位置 | 社区内容在回答中出现的位置（首选推荐 / 对比提及 / 仅提一句） |
| 竞品对比 | 同一关键词下竞品（CentOS、Ubuntu）的提及率 |

### 12.3 关键词维度（按关键词展开）

| 指标 | 说明 |
|------|------|
| 关键词覆盖状态 | 该关键词在各平台的搜索结果汇总 |
| 内容缺口标记 | AI 完全没提到社区 → 标记为"缺口" |
| 改进建议 | 针对该关键词的具体行动项 |

## 十三、GEO 评分体系

### 13.1 单关键词单平台评分（0-10 分）

```
评分 = 提及分 + 引用分 + 准确分 + 排名分

提及分（0-3）：
  0 = 完全未提及
  1 = 仅提及社区名称
  2 = 有实质性描述
  3 = 作为首选/重点推荐

引用分（0-3）：
  0 = 无引用来源
  1 = 引用第三方关于社区的内容
  2 = 引用社区官方内容但非最佳页面
  3 = 引用社区官方文档/博客的精准页面

准确分（0-2）：
  0 = 信息错误或严重过时
  1 = 基本正确但有小问题
  2 = 完全准确且为最新版本

排名分（0-2）：
  0 = 在竞品之后提及或仅做背景提及
  1 = 与竞品并列对比
  2 = 排在竞品之前或作为首选推荐
```

### 13.2 平台评分（0-100）

```
平台评分 = (所有关键词得分之和 / 关键词数 / 10) × 100
```

### 13.3 综合评分（0-100）

```
综合评分 = Perplexity 评分 × 0.35
         + ChatGPT 评分 × 0.40
         + DeepSeek 评分 × 0.25
```

权重依据：ChatGPT 用户基数最大权重最高，Perplexity 引用分析最精确次之，DeepSeek 覆盖国内市场。

### 13.4 评分等级

| 分数区间 | 等级 | 含义 |
|---------|------|------|
| 80-100 | A | GEO 表现优秀，AI 搜索生态覆盖充分 |
| 60-79 | B | 表现良好，部分关键词存在改进空间 |
| 40-59 | C | 表现一般，多个关键词存在内容缺口 |
| 20-39 | D | 表现较差，AI 平台对社区认知不足 |
| 0-19 | F | 基本缺失，需要系统性建设 |

## 十四、改进建议生成

根据评分结果，自动生成分优先级的改进建议。

### 建议示例

```
关键词："CentOS 迁移到 openEuler"
当前评分：4/10
问题诊断：
  - Perplexity：引用了第三方博客而非官方文档
  - ChatGPT：提到了 x2openEuler 工具但信息为旧版本
  - DeepSeek：未提及
改进建议：
  [P0] 更新官方迁移文档，确保包含最新 x2openEuler 版本信息
  [P1] 发布一篇结构化的迁移教程博客（含 FAQ 段落，便于 AI 引用）
  [P2] 在论坛置顶迁移专题帖，增加该话题的内容密度
```

### 优先级规则

- **P0**：评分 0-3 且为高优先级关键词 → 紧急补充内容
- **P1**：评分 4-6 → 优化现有内容
- **P2**：评分 7-8 → 微调优化
- 评分 9-10 → 无需行动，持续监控

## 十六、技术实现方案

采用前后端分离架构：

| 模块 | 方案 |
|------|------|
| 前端 | React / Vue，看板可视化使用 ECharts / Ant Design Charts |
| 后端 | Python（FastAPI / Flask），提供 REST API |
| 数据采集 | Python 脚本调用各平台 API |
| 评分计算 | LLM 对搜索结果做结构化评估，输出各维度分数 |
| 准确率判断 | LLM 自动判断（判断标准待后续讨论确定） |
| 改进建议 | LLM 根据评分和原始搜索结果生成建议 |
| 数据存储 | PostgreSQL（支持多人协作） |
| Excel 导出 | 后端使用 openpyxl / pandas `to_excel()` 生成，前端提供下载接口 |
| 定时任务 | Cron / Celery Beat，每周自动采集并更新评分 |
